# 展示型减负改造设计

## 背景

当前项目底层能力已经比较完整：多 Agent、知识图谱、DAG 编排、报告生成、质量检查、前端展示和测试体系都已经存在。但对外展示时，复杂架构被放得太靠前，普通评委、面试官或第一次打开项目的人需要先理解很多技术概念，才能知道这个工具到底能做什么。

这次改造的目标不是继续增加能力，而是把项目变成更适合作品集和演示的形态：先让人看懂结果，再让技术细节证明结果可信。

## 目标

- 首页和 README 能在 1 分钟内说明项目价值。
- 默认 Demo 流程稳定、简单、容易点击体验。
- 对外主线从“14 个 Agent 协作系统”调整为“自动生成竞品分析报告的 AI 工具”。
- 保留现有后端能力，但减少前端首屏和文档开头的技术负担。
- 让技术亮点服务于可信度，而不是抢占主线。

## 非目标

- 不新增 Agent。
- 不新增数据源。
- 不重构整个后端。
- 不追求所有自由输入场景都在 Demo 中稳定。
- 不删除 DAG、知识图谱、Agent 轨迹、QA 等已有能力。
- 不把首页继续做成复杂控制台。

## 推荐方案

采用“作品集 Demo 化”方案。

项目对外只讲一句话：

> 输入一个产品，系统自动找资料、整理证据，并生成带图表和来源的竞品分析报告。

底层的 14 个 Agent、知识图谱、DAG、缓存、降级和 QA 仍然保留，但不放在第一层解释里。README 和首页前半部分写给普通人，后半部分再给技术读者。

## 默认 Demo 流程

默认流程压成 4 步：

1. 输入分析目标：用户只填目标产品和竞品范围。
2. 确认资料来源：展示官网、GitHub、新闻、社区、应用商店等来源，让用户确认。
3. 生成报告：前端只显示资料收集、结构化分析、报告撰写、质量检查四个阶段。
4. 查看结果：优先展示完整报告、关键图表和来源引用，技术细节放到次级入口。

默认 Demo 使用稳定模板流程，不把 Orchestrator 自由规划作为主入口。Orchestrator 可以保留为高级或旧版选项。

## 预设案例

首页提供 2 到 3 个预设案例，让访客不需要先思考输入内容：

- AI 编程助手竞品分析
- 项目管理工具竞品分析
- 浏览器插件竞品分析

预设案例要优先选择数据源稳定、报告效果清楚、图表容易生成的方向。自由输入仍然可以保留，但不作为第一体验。

## 首页设计

首页从“后台系统控制台”调整为“作品集 Demo 入口”，建议包含：

- 一句话说明：输入一个产品，自动生成有证据、有图表、有结论的竞品分析报告。
- 预设 Demo 按钮：让用户一键开始。
- 四阶段进度：资料收集、结构化分析、报告撰写、质量检查。
- 示例报告预览：展示一小段结论、几张图表和来源数量。
- 技术亮点：多 Agent 分工、知识图谱保存证据、质量检查减少幻觉、任务隔离避免混入旧数据。

Agent 轨迹、DAG 图、知识图谱细节放到“技术细节”区域，不作为首屏主内容。

## README 设计

根目录需要补一个清晰的 README，建议顺序如下：

1. 项目一句话介绍
2. Demo 截图或效果说明
3. 它能解决什么问题
4. 快速运行方法
5. 使用流程
6. 核心能力
7. 技术架构
8. 测试方法
9. 已完成和待办事项
10. 搜索记录

README 前半部分避免堆技术名词，重点说明“用户能得到什么”。技术架构放在后半部分，用来证明项目不是简单提示词包装。

## 架构影响

这次改造不改变核心后端架构。

现有链路保持不变：

知识图谱保存证据和分析节点，Agent 负责采集、分析、写作和检查，DAG 控制执行顺序，API 向前端提供任务、报告、图表和轨迹。

变化主要发生在展示层和文档层：

- 前端默认入口更轻。
- README 补齐并调整叙事顺序。
- 技术细节从主入口后移。
- 默认 Demo 强调稳定模板流程。

## 数据流

推荐展示数据流：

1. 用户选择预设案例或输入分析目标。
2. 系统生成候选来源。
3. 用户确认来源。
4. 后台按稳定模板执行采集、分析、写作和 QA。
5. 知识图谱保存证据、结构化结论和报告片段。
6. 前端展示报告、图表、来源和可选技术细节。

这条数据流要在 README 和首页中用简单语言表达，不展开底层类名和模块名。

## 错误处理

Demo 场景要优先保证“有结果可看”：

- 来源不足时，明确提示“资料较少，报告可信度会下降”。
- 单个来源抓取失败时，不中断整个报告，继续使用其他来源。
- 图表数据不足时，显示文字结论和来源说明，不展示空白图。
- 后台异常时，用非技术语言说明失败原因，例如“资料收集失败，请换一个预设案例或稍后重试”。

## 测试与验收

验收重点放在展示体验：

- README 能说明项目用途、运行方法和演示流程。
- 首页能直接看到预设 Demo 入口。
- 默认流程只暴露 4 个阶段。
- 报告页优先展示报告、图表和来源。
- 技术细节仍然能从次级入口查看。
- 现有核心测试继续通过，前端渲染测试覆盖报告和图表展示。

测试结果对外只汇总通过和失败数量，不逐条展开用例。

## 实施边界

本次改造建议分两步做：

第一步先补 README 和首页叙事，让项目马上变得容易理解。

第二步再整理默认 Demo 流程，把预设案例、四阶段进度和技术细节入口做得更清楚。

不建议在这个阶段继续扩大数据源或新增 Agent，否则会再次增加复杂度，削弱展示效果。
