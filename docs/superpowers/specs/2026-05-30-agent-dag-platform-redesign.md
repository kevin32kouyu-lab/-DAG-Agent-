# 自研 Agent DAG 平台引擎改造设计

日期：2026-05-30

## 1. 背景

当前项目已经具备知识图谱、14 个 Agent、自研 DAG 调度、QA 反馈、报告生成、WebSocket 监控和 React 前端。但整体形态更像技术展示系统：模块很多，能力分散，用户主线不够清晰。

新的方向不是为了单次答辩做演示，而是把项目重塑成一个真实可用的竞品分析产品。第一类目标用户是产品经理和创业者，他们最关心的是能否快速得到一份可信、可读、可导出、可溯源的竞品分析报告。

## 2. 产品目标

第一阶段产品只聚焦一条主线：

用户输入目标产品和竞品，系统自动完成公开信息采集、结构化分析、质检和报告生成，最终输出一份带来源、可信度和导出能力的竞品分析报告。

长期方向是：

- 自研 Agent DAG 平台引擎
- 竞品分析工作流模板
- 一键报告产品体验
- 可扩展到更多行业和更多 Agent 类型

## 3. 非目标

第一阶段不做以下内容：

- 不接入 LangGraph 作为主流程依赖。
- 不做拖拽式工作流编辑器。
- 不让用户手动配置每个 DAG 节点。
- 不追求任意行业自动适配。
- 不把 Agent 数量作为产品卖点。

这些能力可以作为后续扩展，但不进入第一阶段。

## 4. 核心设计判断

项目应把自研 DAG 做成核心能力，而不是把它当成普通任务调度器。

推荐技术主张：

> 系统参考多 Agent 工作流思想，但为了更好支持竞品分析中的证据溯源、QA 打回、节点审计、模板化工作流和报告可信度控制，选择自研 Agent DAG 平台引擎。

这条路线比直接套用通用编排框架更贴合本项目，也能保留当前已经完成的工程资产。

## 5. 后端总体架构

后端分为七层。

### 5.1 工作流模板层

负责定义不同分析场景的标准流程。

第一阶段支持两个模板：

- SaaS 竞品分析模板
- App / 互联网产品分析模板

模板描述内容包括：

- 阶段顺序
- 需要启用的 Agent
- 每个 Agent 的输入要求
- 每个 Agent 的输出 Schema
- 节点依赖关系
- 允许的降级策略
- QA 失败后的打回目标

### 5.2 DAG 编译层

负责把用户输入和工作流模板编译成一张具体任务图。

输入示例：

- 目标产品：Notion
- 竞品：ClickUp、飞书
- 场景：SaaS
- 深度：标准分析

输出是一张可执行 DAG，包含采集、结构化、分析、报告和质检节点。

第一阶段不让 LLM 自由生成整张 DAG。DAG 主要来自确定性模板，只允许少量动态分支，例如 SaaS 场景启用定价分析，App 场景启用社区和留存分析。

### 5.3 调度执行层

负责执行 DAG。

核心职责：

- 判断节点依赖是否满足
- 并发执行可运行节点
- 记录节点状态
- 处理超时
- 处理失败重试
- 执行降级策略
- 支持局部重跑
- 支持断点恢复

节点状态建议统一为：

- pending：等待依赖
- ready：可执行
- running：执行中
- completed：完成
- failed：失败
- degraded：降级完成
- rejected：被质检打回
- rerunning：重跑中

### 5.4 Agent Runtime

负责运行具体 Agent。

每个 Agent 只负责清晰职责，不直接和其他 Agent 自然语言聊天。Agent 通过结构化输入、结构化输出、知识图谱和事件流协作。

Agent Runtime 需要统一处理：

- Prompt 构建
- 工具调用
- 输出校验
- Schema 归一
- StepTrace 记录
- Token 和成本统计
- 错误提示

### 5.5 知识图谱与证据层

知识图谱是系统的可信数据中心。

需要保存：

- 原始网页、新闻、评论、价格信息
- 产品基本信息
- 功能树
- 定价模型
- 用户画像
- 舆情结论
- 市场定位
- SWOT
- 报告段落
- 结论和证据关系
- Agent 生成记录
- QA 校验记录

关键原则：

Agent 不直接依赖其他 Agent 的自然语言输出，而是读取图谱中结构化的中间结果。

### 5.6 反馈控制层

质检 Agent 不能只输出建议，必须能触发系统动作。

QA 失败时需要输出：

- 失败结论 ID
- 问题类型
- 关联证据
- 责任节点
- 建议打回阶段
- 是否允许降级通过

反馈控制层根据 QA 输出决定：

- 局部重跑采集节点
- 局部重跑分析节点
- 只重写报告段落
- 标注低可信并继续输出

第一阶段必须保证 QA 打回是真实生效的，而不是只在报告中展示一段文字。

### 5.7 事件与可观测层

系统需要持续向前端输出事件。

事件包括：

- 任务创建
- DAG 已生成
- 节点开始
- 节点完成
- 节点失败
- 节点降级
- QA 打回
- 局部重跑
- 报告生成
- 成本更新

用户主界面只展示易懂进度；技术详情页展示完整 DAG、Agent 日志、工具调用、Token、耗时、错误和 Trace。

## 6. Agent 体系

Agent 不压缩成 4 个，但按角色组组织。

### 6.1 采集组

职责：发现、采集和整理公开资料。

建议包含：

- 竞品发现 Agent
- 网页采集 Agent
- 评论 / 社媒采集 Agent
- 问卷设计 Agent
- 用户访谈资料整理 Agent

第一阶段重点先保证网页、官网、价格页、公开评论和新闻可用。问卷和访谈先作为结构化资料入口，不要求真实发放问卷。

### 6.2 分析组

职责：把资料转成可比较的业务洞察。

建议包含：

- 功能树分析 Agent
- 定价模型 Agent
- 用户画像 Agent
- 舆情分析 Agent
- 技术栈分析 Agent
- 市场定位 Agent
- SWOT 分析 Agent

### 6.3 报告组

职责：把结构化分析结果生成可读报告。

建议包含：

- 报告撰写 Agent
- 摘要生成 Agent
- 图表数据生成 Agent

报告必须来自图谱中的结构化结果，不能只让 LLM 根据零散上下文直接写全文。

### 6.4 质检组

职责：检查事实、逻辑和来源。

建议包含：

- 事实校验 Agent
- 逻辑一致性 Agent
- 来源覆盖检查 Agent
- 质量评分 Agent

质检结果需要写入图谱，并能触发局部重跑。

## 7. 一键报告流程

第一阶段主流程：

```text
创建任务
→ 选择 SaaS 或 App 模板
→ 编译 DAG
→ 实时采集公开资料
→ 采集失败时使用缓存或备用来源降级
→ 写入知识图谱
→ 多 Agent 结构化分析
→ 生成报告草稿
→ QA 质检
→ 不通过则局部重跑
→ 生成最终报告
→ 支持导出和溯源查看
```

用户看到的是一键报告体验，不需要理解所有技术细节。

## 8. 数据可信度设计

每条关键资料和结论都需要带来源状态。

建议状态：

- realtime：实时采集
- cached：缓存复用
- fallback：备用数据
- failed：采集失败
- verified：已通过质检
- low_confidence：低可信

报告中的关键结论需要能追溯到：

- 原始来源 URL 或资料入口
- 采集时间
- 生成该结论的 Agent
- 使用过该结论的报告段落
- QA 是否通过

## 9. 前端产品体验

第一阶段不做拖拽式 DAG 编辑器。

主界面围绕一键报告：

- 首页：输入目标产品、竞品、场景和分析深度
- 进度页：展示采集、分析、报告、质检阶段
- 报告页：展示正文、图表、来源、可信度和导出按钮
- 溯源页：点击结论查看证据链和 Agent 过程
- 技术详情页：查看完整 DAG、节点状态、日志、成本和重跑记录

普通用户默认只看报告和来源；高级用户可以进入技术详情。

## 10. 第一条稳定样例

第一阶段主样例：

```text
Notion vs ClickUp vs 飞书
```

原因：

- 适合 SaaS 模板
- 资料来源丰富
- 功能、定价、用户群体、生态和市场定位都可比较
- 覆盖中英文资料
- 适合验证实时采集、结构化分析、报告和溯源

## 11. 错误处理与降级

系统需要保证一键报告尽量完成。

错误处理原则：

- 单个来源失败不应导致整条任务失败。
- 采集失败时优先换备用来源。
- 备用来源失败时使用缓存。
- 仍然失败时标注低可信，并继续生成部分报告。
- QA 无法完全确认时，应输出风险提示而不是阻塞报告。
- 节点多次失败后进入 degraded 状态。

用户看到的是可理解提示，例如“部分来源采集失败，已使用缓存资料补充”，而不是技术异常。

## 12. 测试策略

测试分四层。

### 12.1 单元测试

覆盖：

- DAG 模板编译
- 节点状态流转
- 依赖判断
- 降级策略
- QA 打回规则
- Schema 校验

### 12.2 回放测试

使用固定 fixture 验证完整流程，保证不依赖网络也能稳定测试。

### 12.3 真实采集烟雾测试

针对 Notion、ClickUp、飞书跑小规模真实采集，验证公开网页、价格页和来源记录可用。

### 12.4 端到端测试

验证从创建任务到报告导出、溯源查看的完整链路。

## 13. 分阶段实施

### 阶段 1：DAG 平台边界整理

目标：

- 明确模板、编译、调度、反馈、事件的模块边界
- 不改变用户体验
- 不破坏现有测试

### 阶段 2：一键报告模板化

目标：

- 增加 SaaS 和 App 模板
- 固化一键报告主流程
- 支持 Notion vs ClickUp vs 飞书稳定运行

### 阶段 3：反馈闭环强化

目标：

- QA 输出结构化打回指令
- 支持局部重跑
- 前端展示 QA 打回和重跑记录

### 阶段 4：报告可信度升级

目标：

- 报告结论绑定证据
- 展示来源状态和可信度
- 支持导出时保留来源信息

### 阶段 5：产品体验收敛

目标：

- 首页简化为一键报告
- 报告页突出结论、图表和来源
- 技术详情放到高级入口

## 14. 风险与控制

主要风险：

- 自研 DAG 继续膨胀，边界不清
- 实时采集不稳定
- 报告生成仍然过度依赖 LLM 自由发挥
- QA 打回只停留在展示层
- 前端被技术细节淹没

控制方式：

- 第一阶段只支持固定模板和少量动态分支
- 保留回放测试作为稳定基线
- 报告生成必须基于图谱结构化结果
- QA 必须输出可执行的打回目标
- 普通用户界面只展示一键报告主线

## 15. 验收标准

第一阶段完成后，应满足：

- 用户可以创建 SaaS 或 App 竞品分析任务
- 系统可以生成具体 DAG
- DAG 可以稳定执行到最终报告
- Notion vs ClickUp vs 飞书样例可以跑通
- 采集结果、分析结论和报告段落写入知识图谱
- 报告关键结论可以查看来源
- QA 可以触发至少一种真实局部重跑
- 失败时可以降级输出可用报告
- 前端可以查看进度、报告、来源和技术详情

## 16. 自检结果

本设计没有要求推倒现有系统，而是把现有能力重新分层。第一阶段范围聚焦一键报告和模板化 DAG，不包含拖拽编辑器、任意行业适配和 LangGraph 接入。设计中的长期平台化目标通过模板、编译、调度、反馈和事件层逐步支撑，不影响第一阶段稳定交付。
