# 字节跳动 AI 全栈项目挑战赛：项目严格评估报告

> 项目：AI 驱动的竞品分析 Agent 协作系统  
> 评估视角：严格产品经理 + 高级程序员 + 比赛评委  
> 评估依据：课题介绍、评分标准、交付要求，以及当前代码项目实现情况。

---

## 1. 总体评价

这个项目方向与比赛课题高度贴合：通过多 Agent 完成公开信息采集、结构化分析、报告生成、事实校验与反馈闭环，最终输出可追溯的竞品分析报告。

当前项目已经具备较完整的工程原型：

- FastAPI 后端
- React 前端
- DAG 调度
- SQLite 知识图谱
- 14 个 Agent 角色
- Agent 工具系统
- WebSocket 实时进度
- 报告输出
- 测试体系
- 缓存、成本统计、审计、快照、降级等基础设施

但是如果按比赛标准严格评价，目前最大问题是：

> 架构看起来很完整，但产品可信闭环、真实演示稳定性、证据链体验、Agent 协作真实性、交付材料表达还不够强。

项目目前更像一个工程框架完成度较高的 Agent 系统原型，距离比赛现场能打动评委的可演示、可信、可解释、业务价值明显的产品仍有差距。

## 3. 按评分维度评价

### 3.1 多 Agent 协作与输出可信度（35%）

#### 当前优点

项目已经覆盖比赛要求中的多 Agent 架构：

- SourceDiscovery：信息源发现
- Collector：网页/公开资料采集
- DataEnricher：数据结构化增强
- FeatureAnalyzer：功能矩阵分析
- SentimentAnalyzer：用户口碑分析
- PricingAnalyst：定价分析
- TechStackAnalyzer：技术栈分析
- MarketPositionAnalyzer：市场定位分析
- CrossReviewAgent：交叉复核
- SWOTAnalyzer：SWOT 综合
- ReportGenerator：报告生成
- QA_FactCheck：事实检查
- QA_LogicCheck：逻辑检查

同时具备：

- DAG 编排
- Agent 执行器
- ReAct 执行循环
- 知识图谱作为共享状态
- QA 与 CrossReview 反馈机制

#### 主要扣分点

##### 1. Agent 协作“看起来有”，但评委可能看不出“真的协作了”

当前协作主要体现在 DAG 依赖、图谱读写和节点状态变化上。但评委更想看到：

- 采集 Agent 找到了哪些信息
- 分析 Agent 如何使用上游信息
- 质检 Agent 发现了什么问题
- 系统如何打回上游重做
- 重做前后结果如何改善

如果前端只是展示节点状态从 pending 到 completed，容易被认为只是流程图，而不是真正的 Agent 协作。

##### 2. 反馈闭环存在，但可演示性不强

代码里已有 QA 和 CrossReview 反馈逻辑，但比赛要求强调“真实可触发”。需要回答：

- 是否有稳定 Demo 场景能触发？
- 前端是否清楚展示？
- 重做前后结果是否可对比？
- 是否能证明不是写死的？

建议准备一个故意制造错误的 Demo：例如 PricingAnalyst 生成无来源价格结论，QA_FactCheck 检测到缺少证据，系统打回重跑，最终修正或标记低可信。

##### 3. 输出 Schema 有，但业务 Schema 不够显性

项目定义了知识图谱节点和输出 contract，但比赛所说的“竞品知识 Schema”需要在产品和文档里更清晰地展示，例如：

```text
竞品知识 Schema
├── Product
├── FeatureTree
├── PricingModel
├── UserPersona
├── SentimentSummary
├── TechStack
├── MarketPosition
├── SWOT
└── EvidenceChain
```

##### 4. 信息来源完整性还不够显眼

评分要求强调每条分析结论可定位到原始数据源。项目有 SourceInfo、WebPage、trace_upstream 等能力，但报告和前端展示还需要更强：

```text
结论：Figma 在协作设计方面领先
证据：官网功能页 / G2 评论 / ProductHunt 页面
可信度：0.82
```

#### 当前估分

**23 ~ 27 / 35**

若增强证据链、Agent Trace、QA 打回重跑 Demo，可冲到 **30+ / 35**。

### 3.2 技术深度与工程完整度（25%）

#### 当前优点

项目具备较完整的工程体系：

- FastAPI 后端
- React 前端
- WebSocket
- DAG 编排
- 多 Agent
- SQLite 知识图谱
- LLM Gateway
- Semantic Cache
- Cost Tracker
- Audit Logger
- Snapshot
- Task Queue
- Degradation Handler
- Health Check
- PDF / analytics / report
- 测试覆盖

#### 主要扣分点

##### 1. Claude / Anthropic 新模型参数存在兼容风险

当前 LLM Gateway 中 Anthropic 调用统一传 `temperature`。如果 reasoning tier 使用 `claude-opus-4-7` 或未来升级 `claude-opus-4-8`，根据当前 Claude API 规则，Opus 4.7 / 4.8 不接受 `temperature/top_p/top_k`，可能导致 API 400。

建议：

- 对 Opus 4.7 / 4.8 移除 `temperature`
- 默认 reasoning 可升级为 `claude-opus-4-8`
- 增加 `thinking={"type": "adaptive"}`
- Anthropic structured output 使用 `output_config.format`
- OpenAI-compatible 与 Anthropic 参数分开处理

##### 2. 真实运行依赖不明确

当前缺少 `.env.example`，对比赛交付扣分。需要说明：

- `ANTHROPIC_API_KEY`
- `OPENAI_API_KEY`
- `OPENAI_BASE_URL`
- `LLM_DEFAULT_MODEL`
- `TAVILY_API_KEY`
- `TIANYANCHA_TOKEN`
- `RUN_REAL_LLM_TESTS`

以及哪些能力是真实采集，哪些是降级或 demo/mock。

##### 3. 系统生产化不足

当前更适合本地 Demo：

- CORS 完全开放
- 没有鉴权
- 没有用户隔离
- SQLite 单文件
- WebSocket 连接状态在内存中
- Scheduler 是进程内单例
- 缺少 Docker/部署说明

比赛不一定要求生产级，但工程完整度会看这些。

##### 4. DAG 并发和恢复逻辑仍可增强

现有 DAG 调度已经能跑，但建议强化：

- 节点耗时统计
- token 消耗统计
- retry 原因展示
- DAG replay
- 可视化状态回放

##### 5. 工程亮点需要被文档和前端表达出来

项目有缓存、成本、降级、快照、健康检查，但如果不展示，评委很难知道。

#### 当前估分

**18 ~ 22 / 25**

若修复 LLM 参数兼容、补部署说明、强化可观测展示，可冲到 **23+ / 25**。

### 3.3 业务价值与产品体验（20%）

#### 当前优点

项目解决的问题明确：产品团队做竞品分析时，需要经历资料收集、功能矩阵、定价分析、用户口碑、SWOT、报告整理等多个耗时步骤。

系统已经具备：

- 任务创建
- 自动采集
- 多维分析
- 报告生成
- 图表展示
- Trace 查看
- PDF / Markdown / JSON 导出

#### 主要扣分点

##### 1. 产品体验偏工程演示

PM 视角会关注：

- 用户如何选择竞品？
- 能否配置分析维度？
- 能否指定信息源？
- 能否人工审核来源？
- 能否修改报告？
- 能否导出给业务方？
- 能否看到结论为什么可信？

如果这些体验不顺，业务价值会被扣分。

##### 2. 报告质量存在不确定性

Writer fallback 中允许在图谱无数据时使用模型训练知识生成报告。这有利于兜底，但对可信度是风险。比赛强调可追溯，如果无来源内容进入报告，会被质疑为幻觉。

建议：

- 没有证据的数据必须标注“无公开来源验证”
- 无来源结论不得伪装成事实
- 报告每节显示证据充分/不足/需人工确认
- 无来源结论进入低可信区域

##### 3. 缺少可操作建议的产品化呈现

好的竞品分析报告应包含：

- 机会点列表
- 风险列表
- 功能差距矩阵
- 定价对比
- 用户痛点 Top N
- 推荐路线图
- 可信度分数

##### 4. 人工介入体验不足

建议支持：

- 删除不可信来源
- 添加自定义来源
- 要求重新采集
- 修正某个结论
- 重新生成报告
- 比较修正前后

#### 当前估分

**12 ~ 15 / 20**

若强化报告可信度、人工修正、证据链和产品化图表，可冲到 **17+ / 20**。

## 4. P0 必须优先修复的问题

### 4.1 修复 LLM Gateway 的 Claude 参数兼容问题

当前风险：

- reasoning 默认模型可能为 `claude-opus-4-7`
- Anthropic 调用传 `temperature`
- Opus 4.7 / 4.8 不接受该参数

建议：

- 对 Opus 4.7 / 4.8 不传 `temperature`
- 默认 reasoning 改为 `claude-opus-4-8`
- 增加模型能力判断
- Anthropic 使用 `output_config.format`
- OpenAI-compatible 使用 `response_format`

### 4.2 准备稳定 Demo / Replay 模式

不要依赖现场真实爬虫、网络和 API。准备固定 fixture 数据，保证 3 分钟内跑通：

```text
创建任务 → DAG 展示 → Agent 执行 → 证据链 → 报告 → QA 反馈 → 导出
```

### 4.3 报告必须增加证据链和可信度

每个核心结论建议格式：

```text
结论：Figma 在多人协同设计方面领先
可信度：高，0.86
证据来源：官网功能页 / G2 评论 / ProductHunt 页面
```

### 4.4 做出 QA 打回重跑的可见闭环

建议 Demo：

```text
PricingAnalyst 生成无来源价格结论
QA_FactCheck 识别“缺少证据”
系统将 PricingAnalyst 标记 rejected
触发重新分析
重跑后结论被修正或标低可信
```

## 7. 答辩建议强调的亮点

### 亮点 1：不是聊天式 Agent，而是 DAG 协作式 Agent

每个 Agent 有明确职责、输入输出和依赖关系，系统用 DAG 控制任务流，而不是让多个 Agent 自由聊天。

### 亮点 2：知识图谱是单一事实源

Agent 不直接互相对话，而是通过结构化知识图谱共享证据和结论，避免上下文污染，也方便追溯。

### 亮点 3：输出不是纯文本，而是结构化竞品知识

功能、定价、口碑、技术栈、市场定位、SWOT 都写入统一 Schema。

### 亮点 4：有横向 CrossReview 和纵向 QA

CrossReview 检查多个分析 Agent 之间的冲突，QA 检查最终报告事实和逻辑，发现问题后可以打回重跑。

### 亮点 5：可观测和可追溯

每个 Agent 的 prompt、输入、输出、动作、token、成本、读写节点都有 trace。

### 亮点 6：工程化考虑完整

缓存、成本统计、降级、快照恢复、健康检查、WebSocket 实时推送。

## 8. 最终结论

这个项目基础很好，尤其是工程结构和 Agent 架构，已经明显超过普通课程 Demo。

但按比赛标准严格看，它目前最大的问题不是“代码不够多”，而是：

> 可信证据链不够显性，反馈闭环不够可演示，产品体验不够业务化，交付材料不够比赛化，LLM 运行兼容性存在风险。

如果现在直接提交，可能是一个中上水平项目。若集中 2 ~ 3 天优化 P0/P1 项，可以变成一个有竞争力的高分项目。

建议最终答辩主线：

> 我们的系统不是简单让大模型写一份竞品报告，而是用 DAG 编排多个专业 Agent，通过知识图谱共享证据，用 QA 和 CrossReview 做可信闭环，最终输出每条结论都可追溯的结构化竞品分析报告。
